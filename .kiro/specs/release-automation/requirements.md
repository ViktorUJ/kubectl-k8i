# Requirements Document: Release Automation

## Introduction

This document describes the requirements for automating continuous integration (CI) and release (CD) processes for the kubectl-k8i project — a kubectl plugin written in Go that displays Kubernetes node resource information. The automation covers: a CI pipeline on GitHub Actions (linting, testing, building), automatic release creation on git tags via GoReleaser, cross-compilation for 6 platforms, krew manifest updates with actual SHA256 hashes, and a Makefile for local development.

## Glossary

- **CI_Pipeline**: GitHub Actions workflow that performs linting, testing, and building on every push and pull request to the main/master branches.
- **Release_Pipeline**: GitHub Actions workflow triggered when a git tag matching the `vX.Y.Z` format is created, performing build, packaging, and publishing a release on GitHub.
- **GoReleaser**: A tool for automating the build, packaging, and publishing of Go projects (goreleaser.com).
- **Krew_Manifest**: The YAML file `krew-manifest.yaml` describing the plugin for the kubectl krew plugin manager, including download URLs and SHA256 hashes for each platform.
- **Makefile**: A file with targets for local development: build, test, lint, release.
- **Target_Platform**: A combination of operating system and processor architecture for cross-compilation (linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64, windows/arm64).
- **Checksum_File**: The file `checksums.txt` containing SHA256 hashes of all release artifacts, generated by GoReleaser.
- **Golangci_Lint**: A Go linter aggregator (golangci-lint.run) used for static code analysis.
- **Semantic_Version_Tag**: A git tag in the format `vX.Y.Z`, where X is the major version, Y is the minor version, Z is the patch version, conforming to the Semantic Versioning 2.0.0 specification.

## Requirements

### Requirement 1: CI Pipeline — Linting

**User Story:** As a developer, I want the code to be automatically checked by a linter on every push and pull request to maintain a consistent code style and catch errors early.

#### Acceptance Criteria

1. WHEN a commit is pushed to the main branch or a pull request is created targeting the main branch, THE CI_Pipeline SHALL run Golangci_Lint for the entire project codebase.
2. IF Golangci_Lint detects violations, THEN THE CI_Pipeline SHALL fail the linting step with a non-zero exit code and display the list of violations in the log.
3. WHEN Golangci_Lint completes without violations, THE CI_Pipeline SHALL proceed to the next pipeline step.

### Requirement 2: CI Pipeline — Testing

**User Story:** As a developer, I want unit tests to run automatically on every push and pull request to ensure code correctness before merging.

#### Acceptance Criteria

1. WHEN a commit is pushed to the main branch or a pull request is created targeting the main branch, THE CI_Pipeline SHALL run unit tests with the command `go test ./pkg/...`.
2. WHEN unit tests are run, THE CI_Pipeline SHALL include the race detector flag (`-race`) to detect data races.
3. IF at least one unit test fails, THEN THE CI_Pipeline SHALL fail the testing step with a non-zero exit code.
4. WHEN all unit tests pass successfully, THE CI_Pipeline SHALL proceed to the next pipeline step.

### Requirement 3: CI Pipeline — Build

**User Story:** As a developer, I want the project to be automatically built on every push and pull request to verify that the code compiles without errors.

#### Acceptance Criteria

1. WHEN a commit is pushed to the main branch or a pull request is created targeting the main branch, THE CI_Pipeline SHALL build the binary with the command `go build ./cmd/kubectl-k8i/`.
2. IF the build fails with a compilation error, THEN THE CI_Pipeline SHALL fail the build step with a non-zero exit code and display the compilation errors in the log.
3. WHEN the build completes successfully, THE CI_Pipeline SHALL finish the workflow with a success status.

### Requirement 4: Release Pipeline — Tag-Triggered Launch

**User Story:** As a maintainer, I want a release to be automatically created when a git tag in the `vX.Y.Z` format is pushed, so that I do not need to perform manual steps to publish a new version.

#### Acceptance Criteria

1. WHEN a Semantic_Version_Tag in the `vX.Y.Z` format is pushed to the repository, THE Release_Pipeline SHALL start the release creation process.
2. WHEN the Release_Pipeline is started, THE Release_Pipeline SHALL use the Go version specified in the project's `go.mod` file.
3. WHILE the Release_Pipeline is running, THE Release_Pipeline SHALL use the version number from the git tag for naming artifacts and creating the GitHub Release.
4. IF the git tag does not match the `vX.Y.Z` format, THEN THE Release_Pipeline SHALL not be triggered.

### Requirement 5: Cross-Compilation for 6 Platforms

**User Story:** As a user, I want to download a binary for my platform to use the plugin on Linux, macOS, or Windows with amd64 or arm64 architecture.

#### Acceptance Criteria

1. WHEN the Release_Pipeline performs the build, THE GoReleaser SHALL compile binaries for each of the 6 Target_Platforms: linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64, windows/arm64.
2. THE GoReleaser SHALL name the binary `kubectl-k8i` for linux and darwin platforms.
3. THE GoReleaser SHALL name the binary `kubectl-k8i.exe` for the windows platform.
4. THE GoReleaser SHALL use the path `./cmd/kubectl-k8i` as the main package for the build.

### Requirement 6: Artifact Packaging

**User Story:** As a user, I want to download an archive containing the binary and license to install the plugin with a single extraction command.

#### Acceptance Criteria

1. THE GoReleaser SHALL package binaries for linux and darwin platforms into tar.gz archives.
2. THE GoReleaser SHALL package binaries for the windows platform into zip archives.
3. THE GoReleaser SHALL name archives using the template `kubectl-k8i_<os>_<arch>` (e.g., `kubectl-k8i_linux_amd64.tar.gz`, `kubectl-k8i_windows_amd64.zip`).
4. THE GoReleaser SHALL include the LICENSE file in each archive.

### Requirement 7: Checksum Generation

**User Story:** As a user, I want to verify the integrity of a downloaded archive using a SHA256 hash to ensure the file has not been corrupted or tampered with.

#### Acceptance Criteria

1. WHEN GoReleaser completes artifact packaging, THE GoReleaser SHALL generate a Checksum_File with SHA256 hashes of all archives.
2. THE Checksum_File SHALL contain one line per archive in the format `<sha256>  <filename>`.
3. THE Release_Pipeline SHALL include the Checksum_File in the GitHub Release as a separate artifact.

### Requirement 8: GitHub Release Creation

**User Story:** As a user, I want to see a release page on GitHub with a description of changes and download links to easily find the version I need.

#### Acceptance Criteria

1. WHEN GoReleaser completes building and packaging, THE GoReleaser SHALL create a GitHub Release with the version number from the git tag.
2. THE GoReleaser SHALL automatically generate a changelog from commits between the current and previous tags.
3. THE GoReleaser SHALL attach all archives and the Checksum_File to the GitHub Release as downloadable artifacts.

### Requirement 9: Krew Manifest Update

**User Story:** As a maintainer, I want the krew manifest to be automatically updated with actual SHA256 hashes and download URLs after each release, so that krew users receive the correct version.

#### Acceptance Criteria

1. WHEN the Release_Pipeline completes the GitHub Release creation, THE Release_Pipeline SHALL update the Krew_Manifest file, replacing the `sha256` values for each of the 6 Target_Platforms with actual SHA256 hashes from the Checksum_File.
2. WHEN the Release_Pipeline updates the Krew_Manifest, THE Release_Pipeline SHALL replace the download URLs (`uri`) for each of the 6 Target_Platforms, substituting the version number from the git tag.
3. WHEN the Release_Pipeline updates the Krew_Manifest, THE Release_Pipeline SHALL update the `spec.version` field to the value from the git tag.
4. WHEN the Release_Pipeline completes the Krew_Manifest update, THE Release_Pipeline SHALL create a commit with the updated Krew_Manifest and push it to the main branch.

### Requirement 10: GoReleaser Configuration

**User Story:** As a maintainer, I want to have a GoReleaser configuration file in the repository so that build and release settings are versioned and reproducible.

#### Acceptance Criteria

1. THE Release_Pipeline SHALL use the `.goreleaser.yaml` configuration file in the repository root to configure GoReleaser.
2. THE `.goreleaser.yaml` SHALL define a build for the main package `./cmd/kubectl-k8i` with ldflags for embedding the version.
3. THE `.goreleaser.yaml` SHALL define 6 Target_Platforms in the builds section.
4. THE `.goreleaser.yaml` SHALL define archive formats: tar.gz for linux and darwin, zip for windows.
5. THE `.goreleaser.yaml` SHALL define Checksum_File generation with the SHA256 algorithm.

### Requirement 11: Makefile for Local Development

**User Story:** As a developer, I want to use a Makefile with convenient targets for building, testing, and linting to perform common operations with a single command.

#### Acceptance Criteria

1. THE Makefile SHALL provide a `build` target that builds the binary with the command `go build ./cmd/kubectl-k8i/`.
2. THE Makefile SHALL provide a `test` target that runs unit tests with the command `go test -race ./pkg/...`.
3. THE Makefile SHALL provide a `lint` target that runs Golangci_Lint for the entire project codebase.
4. THE Makefile SHALL provide a `test-integration` target that runs integration tests with the command `go test -tags=integration ./test/integration/...`.
5. THE Makefile SHALL provide a `test-e2e` target that runs e2e tests with the command `go test ./test/e2e/...`.
6. THE Makefile SHALL provide a `test-bench` target that runs benchmarks with the command `go test -bench=. ./test/benchmark/...`.
7. THE Makefile SHALL provide a `release-local` target that performs a local release build via GoReleaser with the `--snapshot` flag without publishing.
8. THE Makefile SHALL provide a `clean` target that removes build artifacts (the binary and the `dist/` directory).
9. THE Makefile SHALL use the `build` target as the default goal.

### Requirement 12: Version Embedding in the Binary

**User Story:** As a user, I want to find out the version of the installed plugin to verify that I am using the latest version.

#### Acceptance Criteria

1. WHEN GoReleaser performs the build, THE GoReleaser SHALL pass the version number from the git tag via ldflags into the binary's version variable.
2. WHEN the Makefile executes the `build` target, THE Makefile SHALL pass the version `dev` via ldflags into the binary's version variable.

### Requirement 13: CI/CD Pipeline Security

**User Story:** As a maintainer, I want the CI/CD pipelines to follow security best practices to minimize the risk of build and release process compromise.

#### Acceptance Criteria

1. THE CI_Pipeline SHALL use pinned versions of GitHub Actions (specifying commit SHA) for all third-party actions.
2. THE Release_Pipeline SHALL use pinned versions of GitHub Actions (specifying commit SHA) for all third-party actions.
3. THE Release_Pipeline SHALL use minimal required permissions for GITHUB_TOKEN (contents: write for creating releases).
4. THE CI_Pipeline SHALL use minimal required permissions for GITHUB_TOKEN (contents: read).
